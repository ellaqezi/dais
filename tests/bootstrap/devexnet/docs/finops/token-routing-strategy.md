## loom-reed-light Command Tier Map

This table makes the implicit FinOps control in task sizing explicit.
S/M task sizing directly governs context window size and therefore token cost.

| Command | Suggested Tier | Rationale |
|---------|----------------|-----------|
| `init light workspace` | Haiku | Scaffolding only; deterministic structure, no design judgment |
| `cleanup after <feature/spec>` | Haiku | Maintenance scheduling; low judgment required |
| `implement task <id>` (S task) | Haiku | ≤250 LOC; bounded scope; Haiku sufficient for mechanical synthesis |
| `refine task <id>` | Sonnet | Decomposition and acceptance-criteria authoring require judgment |
| `implement task <id>` (M task) | Sonnet | 251-500 LOC; architecture fit required; Sonnet handles context span |
| `add a spec for <topic>` | Sonnet | Behavior and constraint definition; design thinking needed |
| `plan next work` | Sonnet | Trade-off analysis across multiple backlog directions |
| `stabilize scope <name>` | Sonnet | Resilience and failure-mode reasoning |
| `plan refactors` | Opus | Structural alignment decisions; architectural blast radius |

**Key insight**: LOOM's S/M sizing rule is a FinOps control. A task that exceeds M must be
decomposed — this is not just a scoping rule, it is a cost gate: tasks larger than M require
Opus-class models to reason about reliably, which increases cost significantly.
